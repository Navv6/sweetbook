# SweetBook 페이지 개발 명세 초안

## 문서 목적
이 문서는 SweetBook의 주요 페이지별로 화면 목적, 사용자 플로우, 필요한 상태, 연동 API, 구현 기준을 정리한 개발 명세 초안이다.

- 대상 범위: `app/` 기준 사용자 진입 페이지와 주요 후속 플로우
- 목적: 이후 기능 추가, UI 수정, API 연결 시 판단 기준 제공
- 원칙: "이 페이지가 퍼널에서 어떤 역할을 하는가"를 먼저 정의하고, 그다음 구현을 붙인다

---

## 전체 플로우

```text
/                      랜딩
  -> /studio/new       테마/판형 선택 및 프로젝트 생성
  -> /projects/[id]    편집
  -> /checkout/[id]    출판/견적/주문
  -> /orders/[id]      주문 결과 및 상태 추적
```
서비스 제안기능
 -> /soundtrack/[id]  QR 기반 북 컨셉과 어울리는 음악 생성 >> 구글 suno로 음원생성이나 원하는 문구 tts 생성으로 보고 듣는 Book으로 활용 제안
### 공통 요구사항

- 각 단계는 이전 단계의 맥락을 이어받아야 한다.
- 빈 상태, 오류 상태, 복귀 경로가 항상 있어야 한다.
- `Project` 기반 화면은 가능한 한 동일한 상태 모델을 공유해야 한다.
- 사용자 입장에서는 "내 책을 만드는 연속 과정"으로 보여야 하며, 페이지가 바뀌더라도 흐름이 끊기면 안 된다.

## 1. 랜딩 `/`

### 역할

- 서비스 첫인상 제공
- 포토북 제작 경험에 대한 기대감 형성
- 스튜디오 진입 CTA 제공

### 사용자 목표

- SweetBook이 무엇을 해주는지 빠르게 이해한다.
- 결과물이 어떤 느낌인지 파악한다.
- 제작을 시작할지 결정한다.

### 진입 조건

- 누구나 진입 가능
- 인증, 프로젝트 상태, 환경 의존 없음

### 이탈 경로

- `/studio/new`로 진입
- 페이지 내 소개 섹션 스크롤 탐색

### 핵심 UI 블록

- `Header`
- `HeroSection`
- `DemoSection`
- `Footer`

### 구현 기준

- 첫 화면에서 SweetBook 브랜딩과 CTA가 즉시 보여야 한다. 
참고 https://www.sweetbook.com/

- CTA는 시각적으로 최우선이어야 한다.
- 결과물 프리뷰는 실제 편집 화면과 연결되는 느낌을 줘야 한다.
- 랜딩은 설명 과다보다 전환 우선 구조여야 한다.

### 추후 확장 포인트

- 실제 제작 샘플 더보기
- 사용 후기/사례
- FAQ 또는 제작 방식 요약

## 2. 스튜디오 시작 `/studio/new`

### 역할

- 프로젝트 생성 전 템플릿과 판형을 결정하는 시작점
- 사용자가 "어떤 책을 만들지"를 정하는 단계

### 사용자 목표

- 테마 선택
- 판형 선택
- 생성 전 미리보기 확인
- 프로젝트 생성

### 진입 조건

- 버튼 이용

### 성공 조건

- `/api/catalog`에서 템플릿/판형 데이터를 가져온다.
- 선택한 `templateId`, `bookSpecId`로 프로젝트 생성이 가능하다.
- 생성 성공 시 `/projects/[id]`로 이동한다.

### 의존 상태

- 로컬 상태
  - `templates`
  - `bookSpecs`
  - `selectedTemplateId`
  - `selectedBookSpecId`
  - `templateSummary`
  - `catalogNotice`
  - `isCatalogLoading`
  - `isGenerating`

### 연동 API

- `GET /api/catalog`
- `POST /api/projects`

### 주요 컴포넌트

- `TemplateSelector`
- `BookSpecSelector`
- `PreviewPanel`
- `StepIndicator`

### 구현 기준

- 좌측은 선택, 우측은 결과 미리보기라는 구조를 유지한다.
- 사용자는 여기서 복잡한 편집을 하지 않는다.
- 카탈로그 실패 시 "왜 지금 선택할 수 없는지"가 보여야 한다.
- 선택 상태가 없으면 생성 버튼이 무의미하게 동작하지 않아야 한다.
- 생성 이후에는 추가 확인 단계 없이 바로 편집기로 이동한다.

### 예외/에러 상태

- 카탈로그 로드 실패
- 템플릿 없음
- 지원 판형 없음
- 프로젝트 생성 실패

### 추후 확장 포인트

- 추천 템플릿 큐레이션 강화
- 최근 작업 재개

## 3. 편집기 `/projects/[id]`

### 역할

- 자동 생성된 포토북을 사용자가 실제 결과물로 다듬는 핵심 작업 공간

### 사용자 목표

- 책 제목 수정
- 섹션 구조 파악
- 페이지별 파라미터 수정
- 이미지 추가 및 교체
- 자유 레이아웃 편집
- 전체 미리보기 확인
- 결제로 이동

### 진입 조건

- `project.id`가 존재해야 한다.
- `useProjectStore` 또는 서버 재주입 경로에서 프로젝트를 읽을 수 있어야 한다.

### 성공 조건

- 현재 프로젝트의 섹션/페이지 구조가 보인다.
- 선택한 페이지를 바로 수정할 수 있다.
- 변경 내용이 스토어에 반영된다.
- 사용자는 편집 맥락을 잃지 않고 `/checkout/[id]`로 이동할 수 있다.
- 자유 편집 기능과 프리뷰에서 책이 동일한 상태를 유지하여야한다.
- 최소 주문 가능 페이지수를 충족하여야 주문페이지로 넘어갈수있다.

### 의존 상태

- 전역 상태
  - `projects[projectId]`
  - `generatedSections`
  - `estimate`
- 로컬 상태
  - `selectedSectionId`
  - `selectedFieldKey`
  - `titleDraft`
  - `isPreviewOpen`
  - `isEstimating`

### 주요 액션

- `replaceSections`
- `duplicateSection`
- `deleteSection`
- `updatePageParameters`
- `updatePageLayoutOverride`
- `updateProjectTitle`
- `setEstimate`

### 연동 API

- `POST /api/projects/[id]/layout`
- `POST /api/projects/[id]/estimate`

### 주요 컴포넌트

- `SectionList`
- `PageCanvas`
- `InspectorPanel`
- `BookPreviewModal`
- `StepIndicator`

### 구현 기준

- 좌측: 구조 탐색
- 중앙: 실제 편집 캔버스
- 우측: 필드/속성 편집
- 현재 선택된 섹션/페이지/필드가 명확해야 한다.
- 제목, 판형, 템플릿, 페이지 수 같은 제작 메타정보가 상단에 노출되어야 한다.
- 최소 페이지 규칙은 판형별로 동작해야 한다.
- 자유 편집 결과는 실제 publish 데이터(`layoutOverrides`)로 이어져야 한다.
- 페이지/프로젝트 없음, 스키마 없음 같은 경우 복귀 CTA가 있어야 한다.
- 자유 편집 레이아웃은 나의 템플릿에 저장되어 불러올수있다.
- 내지 갤러리형이나 전체페이지를 사진을 첨부하는 페이지는 여러장의 사진을 갤러리형으로 자동배치하거나 한장의 사진을 이미지 레이아웃을 채운다. 단일 또는 다수의 사진 첨부가능해야함


### 예외/에러 상태

- 프로젝트 없음
- 렌더 가능한 스키마 없음
- 견적 계산 실패
- 섹션 재정렬 실패

### 추후 확장 포인트

- Undo/Redo
- 커스텀 템플릿 불러오기 개선
- 프로젝트 목록/복귀 허브 연결

## 4. 결제/주문 `/checkout/[id]`

### 역할

- 편집 결과를 실제 출판/주문으로 확정하는 최종 실행 단계

### 사용자 목표

- 배송 정보 입력
- 수량 선택
- 최종 미리보기 확인
- 출판 및 주문 실행

### 진입 조건

- 유효한 `project.id`
- 편집 완료 상태의 프로젝트

### 성공 조건

- 필요 시 publish를 먼저 수행한다.
- 견적을 계산한다.
- 주문을 생성한다.
- 주문 생성 후 `/orders/[id]`로 이동한다.

### 의존 상태

- 전역 상태
  - `project`
  - `project.estimate`
  - `project.order`
  - `project.sweetbookBookUid`
- 로컬 상태
  - `shipping`
  - `quantity`
  - `isSubmitting`
  - `isPreviewOpen`

### 연동 API

- `POST /api/projects/[id]/publish`
- `POST /api/projects/[id]/estimate`
- `POST /api/projects/[id]/order`

### 구현 기준

- 이 페이지는 단순 결제 폼이 아니라 `publish -> estimate -> order` 실행 화면이다.
- 이미 출판된 프로젝트는 재출판하지 않는다.
- 배송 정보와 주문 요약은 시각적으로 분리한다.
- 사용자는 현재 어떤 단계가 진행 중인지 이해할 수 있어야 한다.
- 오류 메시지는 최소한 `출판 실패`, `견적 실패`, `주문 실패` 수준으로 구분되어야 한다.

### 예외/에러 상태

- 프로젝트 없음
- publish 실패
- estimate 실패
- order 실패

### 추후 확장 포인트

- 실제 PG 연동
- 배송지 저장
- 쿠폰/할인

## 5. 주문 확인 `/orders/[id]`

### 역할

- 주문 완료 확인
- 주문 이후 상태 추적
- 후속 경험 진입 허브

### 사용자 목표

- 주문 접수 여부 확인
- 주문 UID/책 UID 확인
- 배송 정보 재확인
- webhook 이벤트 기반 상태 추적

### 진입 조건

- `project.id` 존재
- 주문 생성 이후 진입 권장

### 성공 조건

- 현재 주문 정보가 보인다.
- 로컬 상태만이 아니라 서버 기준 최신 상태로 재동기화된다.
- 주문 이후 상태 변화가 누적 표시된다.

### 의존 상태

- 전역 상태
  - `project.order`
  - `project.webhookEvents`
  - `project.sweetbookBookUid`
- 로컬 상태
  - `isHydrating`

### 연동 API

- `GET /api/projects/[id]`

### 구현 기준

- 1회성 완료 화면이 아니라 상태 확인 허브여야 한다.
- 재진입했을 때도 주문이 보존되어야 한다.
- webhook 이벤트는 추후 운영/고객 대응 관점에서도 읽기 쉬워야 한다.


### 예외/에러 상태

- 초기 hydration 중
- 프로젝트는 있으나 주문 없음
- 서버 동기화 실패

### 추후 확장 포인트

- 배송 단계별 타임라인 UI
- 송장 링크
- 주문 취소/문의 기능

## 6. 사운드트랙 `/soundtrack/[id]`

### 역할

- 완성된 포토북의 감정선을 디지털 오디오 경험으로 확장하는 페이지

### 사용자 목표

- QR 유입 후 즉시 감상
- 책 제목과 연결된 음악 경험
- QR 기반 북 컨셉과 어울리는 음악 생성 >> 구글 suno로 음원생성이나 원하는 문구 tts 생성으로 보고 듣는 Book으로 활용 제안
- 현재 mp3파일 첨부로 동일 적용
### 진입 조건

- QR 또는 직접 진입
- 가능하면 `project`가 있으면 좋지만, 없어도 완전한 빈 화면은 피해야 한다

### 성공 조건

- 오디오 생성/조회 시도
- 재생 가능 시 바로 플레이어 제공
- 불가 시 상태 메시지 제공

### 의존 상태

- 전역 상태
  - `projects[projectId]`
- 로컬 상태
  - `audioUrl`
  - `loading`
  - `message`

### 연동 API

- `POST /api/projects/[id]/music`

### 구현 기준

- 단일 목적 화면으로 유지한다.
- 모바일 QR 진입 기준으로 단순하게 설계한다.
- `loading`, `audioUrl 있음`, `audioUrl 없음`, `project 없음` 상태를 분리한다.
- 책 제목이 없더라도 query param 등으로 최소 표시값을 제공해 빈 화면처럼 보이지 않게 한다.

### 예외/에러 상태

- 프로젝트 없음
- 음악 생성 실패
- 음악 미제공 상태

### 추후 확장 포인트

- 플레이리스트/복수 트랙
- 배경 비주얼 커스터마이징
- 공유 링크

## 구현 공통 규칙

### 상태 관리

- 클라이언트 편집 상태는 `useProjectStore`를 기준으로 유지한다.
- persist 저장 시 data URL 과대 저장 이슈를 항상 점검한다.
- 프로젝트 관련 페이지는 `Project` 단일 모델 기준으로 연결한다.

### 라우트/서버 연동

- App Router 기준 구현
- API 라우트는 thin wrapper 유지
- 서버 비즈니스 로직은 가능하면 `lib/api.ts`에 집중
- Sweetbook api 개발문서 이용 및 기준 준수
- 개발과제에 제시된 내용 확인 필수

### UX 규칙

- 각 페이지는 현재 단계와 다음 행동이 분명해야 한다.
- 실패 상태에는 원인과 복귀 경로를 같이 준다.
- 단계형 서비스이므로 메타 정보와 진행 상태 표시가 중요하다.

### 문서 운영 규칙

- 새 기능을 추가할 때는 먼저 "어느 페이지 역할을 강화하는지"를 이 문서에 연결한다.
- 기능 단위가 아니라 페이지 책임 단위로 정리한다.
- API 명세서와 분리하되, 화면에서 직접 사용하는 엔드포인트는 여기에도 남긴다.

## 사운드트랙 보충 메모

### 기능 의도

- 사운드트랙은 단순 오디오 재생 기능이 아니라, QR 기반 북 컨셉과 연결된 후속 감상 경험으로 본다.
- 제품 방향은 포토북을 "보는 책"에서 "보고 듣는 Book"으로 확장하는 것이다.
- 사용자는 주문 완료 이후 QR을 통해 책의 분위기에 맞는 음악이나 음성을 이어서 경험한다.

### 제안하는 장기 방향

- 책 컨셉과 어울리는 음악 생성
- Google Suno 등 외부 음원 생성 서비스와의 연동 검토
- 사용자가 원하는 문구를 TTS로 생성해 재생
- 제목, 헌사, 특정 본문 문장을 들려주는 방식으로 확장

### 현재 구현 원칙

- 현재는 개인화 생성 기능까지 구현하지 않는다.
- 폴더 속 MP3 파일을 공통 리소스로 재생하는 수준으로 동일 적용한다.
- 즉, 지금 단계의 사운드트랙은 컨셉 검증용이며, 후속 구현 방향을 문서로 먼저 고정하는 목적이 크다.

### 이후 구현 포인트

- 프로젝트별 컨셉 문장 또는 메타데이터를 바탕으로 음악 프롬프트 생성
- 음원 생성 결과물 저장 위치와 매핑 정책 정의
- TTS 입력 문구 규칙 정의
- 주문 완료 페이지 QR과 프로젝트별 오디오 자산 연결
- 공통 MP3 재생 구조에서 프로젝트별 리소스 재생 구조로 전환
